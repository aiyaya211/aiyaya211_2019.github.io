不过Object​.define​Property()也存在很多缺陷，但是对这个属性方法不熟悉的我，阅读源码时遇到了很多不便利，查了网上资料，自己整理一下Object​.define​Property()的用法
Object​.define​Property()用于在一个对象上新增属性或者修改已有属性，举个🌰

对象的属性描述符（就是对象属性的属性）一般分为两种，一种就是数据属性描述符（属性的读写属性一类），另一种就是存取描述符（给属性修改值或者获取属性值），
主要看存取描述符：
get

一个给属性提供 getter 的方法，如果没有 getter 则为 undefined。当访问该属性时，该方法会被执行，方法执行时没有参数传入，但是会传入this对象（由于继承关系，这里的this并不一定是定义该属性的对象）。
默认为undefined。

set

一个给属性提供 setter 的方法，如果没有 setter 则为 undefined。当属性值修改时，触发执行该方法。该方法将接受唯一参数，即该属性新的参数值。
默认为 undefined。

然后再看下Object​.define​Property()的缺陷: 在对象上新增和删除属性都无法被追踪到。

20191001 火曜日 更新
纸上得来终觉浅 昨天在项目中遇到了关于这个object.defineProperty()中的这个坑。
需求内容是：根据后端传过来的数据渲染表格数据，然后当再次从输入框输入相应查询数据时，将对应行数据的状态status状态置为正常。
按照需求，当第二次输入查询对应内容的时候，我将tableData数组中对应的那一条对象数据status属性改成了正常,打印数据也是对的，但是表格UI界面没有任何变化，我就纳闷了….
后来仔细看了接口数据，发现后段给的数据中，也就是我一开始拿到的tableData中根本没有status字段，相当于触及了object.defineProperty()无法侦听对象的属性新增或者删除这个情况。bug算是找到出处了，接下来就是要解决，baidu了一波，发现了vue中自带解决这个问题的方法，使用
Vue.set( target, propertyName/index, value )

向响应式对象中添加一个属性，并确保这个新属性同样是响应式的，且触发视图更新。它必须用于向响应式对象上添加新属性，因为 Vue 无法探测普通的新增属性 (比如 this.myObject.newProperty = ‘hi’)

或者不用这个方法的话，就是一开始就给数据加上这个status属性，赋值为空的字符串也行，遇到bug不可怕，遇不到bug才可怕吧。

JavaScript是一门单线程语言，在w3的官网，对于事件循环的定义是：

To coordinate events, user interaction, scripts, rendering, networking, and so forth, user agents must use event loops as described in this section. There are two kinds of event loops: those for browsing contexts, and those for workers.

放在浏览器中执行一下这段代码，执行结果是

JavaScript中任务分为同步任务和异步任务，同步任务指的是立即执行的任务，异步任务指的是相对比较耗时的任务，像请求后台资源等操作，同步任务一般都在JavaScript的主线程中，异步任务则在任务队列中等待，当主线程中的同步任务执行完成，依次执行任务队列中的事件。
异步任务又分为微任务和宏任务，这两种任务分别在各自的队列中，当主线程中的同步任务执行完之后，再依次从队列中取出宏任务和微任务执行，再执行宏任务之前需要先清空微任务队列。
那，微任务和宏任务又包括哪些呢？

宏任务（micro-task）主要是：script代码段、setTimeout、setInterval、Promise的构造函数是同步的、setImmediate、I/O、UIrendering
微任务（macro-task）主要是：Promise的回调、process.nextTick

就用上面的逻辑来看下这段代码的执行逻辑

  1. 代码进来遇到第一个setTimeout是异步任务，进入宏队列，同时又有一个promise.then（）是微任务，进入微队列
  2. 继续往下走遇到第二个setTimeout也是异步任务，它执行一个s2，进入宏队列，然后p2进入微队列==这里注意是有先后顺序的区分的==
  3. s1先从宏队列出来进入主线程栈，再出栈，所以先打印s1
  4. 往下走宏队列里面是s2，但是记住执行宏任务前先清空微任务队列，所以接下去是p1，再之后s2和p2按照顺序

所以最后的打印顺序是 s1->p1->s2->p2
